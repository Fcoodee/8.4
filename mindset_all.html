CASE 1 · الفكرة الأصلية للمطور
صفحة هبوط / إقلاع فقط لعملية القبول والتسجيل
قرار الانطلاق
مركز صناعة القرار يوافق فقط على الفكرة الأصلية للمطور، دون التوسّع لإعادة تصميم الموقع.
الأطراف المنفّذة
المطور (تطوير) + إدارة القبول والتسجيل (تشغيل) + ShareEdu (نطاق فرعي فقط)
النطاق
صفحة واحدة على نطاق فرعي (مثل portal.daralforsan.edu.sa) لاستقبال ملف المتقدم + لوحة تحكم داخلية بسيطة لفريق القبول (ثلاث مراحل: استقبال / مراجعة / قبول).
الحزمة التقنية
Next.js + Tailwind + PostgreSQL (Supabase Free) + WhatsApp API. بلا VR، بلا chatbot ذكي في المرحلة الأولى.
الجدول الزمني
2-4 أسابيع من قرار الانطلاق حتى الإطلاق التجريبي.
التكلفة
الإعداد: ~3,000 SAR · التشغيل: 300-400 SAR/شهر (WhatsApp API فقط)
المخاطر
فقدان فرصة تطوير الموقع الرسمي · بقاء الموقع الأم على بنيته الحالية.
معيار النجاح
تسهيل عمل فريق القبول لموسم 1447-1448هـ / 2026-2027م + استقبال أول 100 طلب تسجيل رقميًا.
▶ خط سير العمليات
  1. الحصول على نطاق فرعي من ShareEdu (webhook أحادي)
  2. بناء صفحة الهبوط + لوحة التحكم في 2-3 أسابيع
  3. تدريب فريق القبول (يوم واحد)
  4. الإطلاق التجريبي + مراقبة الأداء لمدة شهر
  5. تقييم التوسّع في مواسم لاحقة بناءً على النتائج
⏱ 2-4 أسابيع 💰 ~3,000 SAR إعداد 👤 المطور فقط

CASE 2 · رؤية قسم الاتصال المؤسسي كاملة
حالة الموافقة على مبادرة الاتصال المؤسسي الشاملة
قرار الانطلاق
مركز صناعة القرار يعتمد مقترح قسم الاتصال المؤسسي بالكامل، ويكلّفهم بتنفيذه.
الأطراف المنفّذة
قسم الاتصال المؤسسي (قيادة) + وكالة تطوير خارجية (تنفيذ) + مصورون / فريق VR + ShareEdu (تعاقد إضافي محتمل)
النطاق
إعادة تصميم كامل الموقع + جولات VR 360° للفرعين + chatbot ذكي + WhatsApp API + subdomains لكل فرع + معرض صور احترافي.
الحزمة التقنية
Next.js + Firebase + Vercel Pro + Landbot + Kuula. حزمة اشتراكات حديثة تعمل في السحابة.
الجدول الزمني
3-6 أشهر من قرار الانطلاق حتى الإطلاق الرسمي.
التكلفة
الإعداد: 15,000 SAR · التشغيل: 500 SAR/شهر · 5 سنوات: 45,000 SAR
المخاطر
Vendor lock-in بـFirebase · تكلفة الاشتراكات المتراكمة · عدم توافق تقني مع بيئة ShareEdu الحالية · فقدان النموذج الأولي المرفوع.
معيار النجاح
إطلاق موقع تسويقي حديث بمعايير عالمية + تجربة رقمية متكاملة لأولياء الأمور.
▶ خط سير العمليات
  1. توقيع عقد مع وكالة التطوير المختارة
  2. جولة تصوير + إنتاج محتوى VR (شهر)
  3. بناء الحزمة التقنية على Firebase (شهرين)
  4. ربط WhatsApp API + chatbot + اختبار (شهر)
  5. ترحيل بيانات ShareEdu أو التكامل معها (شهر)
  6. الإطلاق + التدريب + التسليم
⏱ 3-6 أشهر 💰 15,000 إعداد + 500/شهر 👤 قسم الاتصال + وكالة ⚠ تعارض مع ShareEdu

CASE 3 · ShareEdu توسّع خدماتها
حالة العمل على البنية التحتية الخلفية الحالية ShareEdu
قرار الانطلاق
مركز صناعة القرار يقرر البناء على بنية ShareEdu القائمة، وتوسيع عقدها لتشمل التطوير الجديد.
الأطراف المنفّذة
ShareEdu منفردةً، بتوسيع عقدها التعاقدي.
النطاق
تحسينات محدودة على النظام الحالي: نموذج تسجيل + إشعارات + تقارير أساسية. ضمن بيئة Classic ASP + Access DB.
الحزمة التقنية
Classic ASP + Access DB (البيئة الحالية) + إضافات على النظام القائم.
الجدول الزمني
غير محدد — يعتمد على أولويات ShareEdu وطاقتهم التقنية.
التكلفة
مبلغ إضافي على العقد القائم (غير معلن مسبقًا) — يُقدّر عادةً بين 8,000 – 20,000 SAR سنويًا.
المخاطر
قيود تقنية موروثة (Access محدود بـ2GB وبـ10-15 مستخدمًا متزامنًا) · محدودية الابتكار · عدم دعم واجهات حديثة (WhatsApp API الرسمي، chatbot ذكي).
معيار النجاح
استقرار وتوافر تحسينات صغيرة ضمن حدود بيئة العمل الحالية.
▶ خط سير العمليات
  1. مفاوضات مع ShareEdu على نطاق التوسّع والتكلفة
  2. توقيع ملحق تعاقدي
  3. ShareEdu ينفّذ حسب جدولها الداخلي
  4. الاعتماد على تسليمها بلا رقابة تقنية داخلية للمدارس
  5. الإطلاق حسب جاهزيتهم
⏱ غير محدد 💰 حسب التفاوض 👤 ShareEdu فقط ⚠ قيود Access

CASE 4 · المطور ينفّذ رؤية شاملة بأدوات مفتوحة
حالة السماح للمطور بتنفيذ الرؤية الشاملة لقسم الاتصال
قرار الانطلاق
مركز صناعة القرار يعتمد الرؤية الشاملة (كل ما اقترحه قسم الاتصال) لكنه يُكلّف مبادرة #التحول_الرقمي_لمدارس_دار_الفرسان_2027 (المطور) بتنفيذها.
الأطراف المنفّذة
المطور (تنفيذ تقني) + قسم الاتصال (شريك محتوى وهوية بصرية) + ShareEdu (نطاقات فرعية / تكامل)
النطاق
نفس نطاق السيناريو 2، لكن باستخدام أدوات مفتوحة المصدر بدل الاشتراكات التجارية.
الحزمة التقنية
Next.js + PostgreSQL + Cloudflare Pages + Marzipano.js (VR مفتوح) + LangChain.js + WhatsApp API + OpenAI/Claude API.
الجدول الزمني
2-4 أشهر (أسرع من السيناريو 2 بفضل غياب مفاوضات الوكالة الخارجية).
التكلفة
الإعداد: 5,000 SAR · التشغيل: 500 SAR/شهر · 5 سنوات: 35,000 SAR · توفير: 10,000 SAR
المخاطر
حساسية سياسية مع قسم الاتصال المؤسسي (شعور بمصادرة المبادرة) · عبء تنفيذي كبير على المطور · ضرورة التنسيق المستمر.
معيار النجاح
تحقيق نفس أهداف قسم الاتصال + توفير 22% من الميزانية + مرونة أعلى.
▶ خط سير العمليات
  1. إصدار قرار إداري بتكليف مبادرة #التحول_الرقمي_لمدارس_دار_الفرسان_2027
  2. اجتماع تنسيقي مع قسم الاتصال لتحديد الهوية والمحتوى
  3. بناء الواجهات + الطبقة الخلفية + التكامل مع ShareEdu
  4. جولة تصوير مشتركة + إنتاج VR
  5. ربط WhatsApp + chatbot + اختبار
  6. الإطلاق + التسليم للفرق التشغيلية
⏱ 2-4 أشهر 💰 5,000 إعداد + 500/شهر 👤 المطور + الاتصال ⚠ حساسية سياسية

CASE 5 · الشراكة الهجينة — الحل التوافقي
حالة تقاطع العمل للمطور مع البنية التحتية الخلفية ShareEdu
قرار الانطلاق
مركز صناعة القرار يعتمد نموذج شراكة بين المطور و ShareEdu — كلٌّ منهما في نطاق تخصّصه.
الأطراف المنفّذة
المطور (الواجهات الأمامية) + ShareEdu (البنية الخلفية والـ APIs) + قسم الاتصال (المحتوى)
النطاق
واجهات جديدة يبنيها المطور تعمل فوق بنية ShareEdu الحالية عبر طبقة APIs توفّرها ShareEdu.
الحزمة التقنية
Frontend: Next.js من المطور · Backend APIs: تُطوّرها ShareEdu فوق Classic ASP · التكامل عبر REST JSON.
الجدول الزمني
6-10 أسابيع (يعتمد على استجابة ShareEdu في تطوير الـ APIs).
التكلفة
تكلفة المطور (0 SAR تطوير) + تكلفة إضافية على عقد ShareEdu (لبناء APIs) — يُقدّر ~3,000-5,000 SAR إضافي مرة واحدة.
المخاطر
الاعتماد على تجاوب ShareEdu وسرعة إنجازهم · قد يحتاج مفاوضات مطوّلة · قد لا تُطوّر APIs بجودة كافية.
معيار النجاح
تكامل حقيقي بين الطرفين + احترام لأدوار كل شريك + بنية قابلة للتوسّع لاحقًا.
▶ خط سير العمليات
  1. اجتماع تقني ثلاثي (المطور + ShareEdu + الاتصال) لتحديد الأدوار
  2. ShareEdu تُطوّر مجموعة APIs محدّدة (تسجيل، رفع مستندات، حالة الطلب)
  3. المطور يبني الواجهات ويختبر التكامل مع الـ APIs
  4. اختبار مشترك على بيئة تجريبية
  5. الإطلاق + المتابعة المشتركة للأداء
⏱ 6-10 أسابيع 💰 مشترك 👤 شراكة ثلاثية

CASE 6 · الحالة الحرجة — الجمود التعاقدي
حالة تمسّك ShareEdu ببنود عقدها ورفض أي تغيير خارجي
قرار الانطلاق
ShareEdu ترفض السماح للمطور بنشر صفحة الهبوط، وترفض تنفيذ قسم الاتصال لخطة التطوير الشاملة، وترفض أي تعديل هيكلي خارجي على الدومين — بحجة أن نظامها الإلكتروني القائم يتأثّر بأي تغيير خارجي، وأن اتفاقيتها التعاقدية تُخوّلها إدارة كامل البنية التحتية للدومين حتى نهاية العقد.
الأطراف المنفّذة
ShareEdu فقط، بلا سلطة عملية للمطور أو قسم الاتصال في الفترة الحالية.
النطاق
لا تطوير جديد على الدومين الحالي. أي تطوير جديد يبقى مؤجّلاً حتى انتهاء عقد ShareEdu.
الحزمة التقنية
استمرار البنية الحالية (Classic ASP + Access) دون تغيير.
الجدول الزمني
انتظار انتهاء عقد ShareEdu (يُفترض 12 شهرًا أو أكثر حسب تاريخ التوقيع).
التكلفة
مالية: 0 SAR إضافية. تكلفة الفرصة: عالية — استمرار الأعباء الإدارية على فريق القبول لموسم كامل.
المخاطر
تدهور تجربة أولياء الأمور · استمرار العمل الورقي · خسارة تنافسية أمام مدارس أخرى ذات بوابات رقمية · إحباط الكوادر الداخلية.
إجراءات الاستعداد (Mitigation)
  1. مراجعة قانونية للعقد — التحقق من صياغة نصوص الحصرية الفعلية (قد لا تكون قاطعة كما يُدّعى).
  2. تفاوض على استثناءات — كصفحة على نطاق منفصل تمامًا (github.io مثلاً).
  3. خطة انتقالية مسبقة — لاستبدال ShareEdu فورًا عند انتهاء العقد.
  4. تجهيز النموذج محليًا بلا نشر — الاستعداد للحظة الانتقال.
معيار النجاح
توثيق شامل للوضع القانوني الحالي + خارطة انتقالية جاهزة للتنفيذ الفوري عند نهاية العقد.
▶ خط سير العمليات (وضع الترقّب)
  1. الحصول على نسخة كاملة من عقد ShareEdu لمراجعته قانونيًا
  2. تحديد تاريخ انتهاء العقد بدقة
  3. بناء نموذج المطور محليًا بلا نشر (استعداد صامت)
  4. وثيقة استراتيجية للانتقال السلس عند نهاية العقد
  5. تحسينات إجرائية داخلية غير رقمية لتخفيف العبء المؤقت على فريق القبول
  6. عند نهاية العقد → تفعيل السيناريو 4 أو 5 مباشرة
⏱ انتظار نهاية العقد 💰 0 SAR مالية · عالية فرصية 👤 ShareEdu (بلا خيارات) ⚠ الحالة الأشد خطورة

🎯 خلاصة خارطة 8.3

السيناريوهات الستة أعلاه تُشكّل معًا كامل الفضاء الممكن لمستقبل هذه المبادرة. أي قرار من مركز صناعة القرار سيقع بالضرورة ضمن أحدها.

الأفضل عمليًا: السيناريو 1 (سريع الإنجاز) أو السيناريو 5 (شراكة توافقية).
الأخطر: السيناريو 6 (الجمود التعاقدي) — يجب الاستعداد له صامتًا في كل الأحوال.
الأقلّ جدوى ماليًا: السيناريو 2 (رغم أنه الأكثر رواجًا في الطرح).

كل الأقسام التقنية التالية (قاعدة البيانات، بنية التكامل، مخطّط الـ Schema) تعمل كأدوات تنفيذ لأيٍّ من السيناريوهات 1 و 4 و 5 — أي حيث يكون للمطور دور فعلي في التنفيذ.

— الجزء التقني —

تفاصيل قاعدة البيانات، بنية التكامل، مخطّط الـ Schema، والحزمة التقنية — أدوات تنفيذية للسيناريوهات 1، 4، 5.

1. لماذا نناقش قاعدة البيانات أصلًا؟

قواعد البيانات هي نقطة التقاطع الحرجة بين المكوّنات الثلاثة (المطوّر، قسم الاتصال، ShareEdu). كل مكوّن يتعامل معها بطريقة مختلفة، وأي خيار نتخذه يؤثّر على:

  • التكلفة الشهرية المتكرّرة (اشتراكات vs مجاني)
  • مرونة التكامل مع بنية ShareEdu القائمة
  • القدرة على النمو مستقبلًا دون إعادة بناء
  • ملكية البيانات (لدى المدارس vs لدى مزوّد خارجي)
  • الالتزامات القانونية (خصوصية بيانات الطلاب — قانون حماية البيانات السعودي PDPL)

-- مخطّط قاعدة بيانات بوابة القبول والتسجيل

CREATE TABLE branches (
  id SERIAL PRIMARY KEY,
  code VARCHAR(20) UNIQUE NOT NULL, -- 'salamah', 'salihiyah'
  name_ar VARCHAR(100) NOT NULL,
  gender VARCHAR(10) NOT NULL, -- 'boys', 'girls', 'mixed'
  is_new BOOLEAN DEFAULT FALSE,
  contact_whatsapp VARCHAR(20)
);

CREATE TABLE tracks (
  id SERIAL PRIMARY KEY,
  code VARCHAR(30) UNIQUE NOT NULL, -- 'ahli', 'diploma', 'special'
  name_ar VARCHAR(100) NOT NULL,
  description_ar TEXT
);

CREATE TABLE applications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  reference_number VARCHAR(20) UNIQUE NOT NULL,
  branch_id INT REFERENCES branches(id),
  track_id INT REFERENCES tracks(id),
  student_data JSONB NOT NULL, -- name, birthdate, grade, etc.
  guardian_data JSONB NOT NULL, -- name, phone, email, etc.
  documents JSONB DEFAULT '[]', -- array of uploaded document refs
  stage VARCHAR(20) NOT NULL DEFAULT 'received', -- received/review/accepted
  status VARCHAR(20) NOT NULL DEFAULT 'active',
  discount_applied BOOLEAN DEFAULT FALSE,
  discount_percentage INT DEFAULT 0,
  submitted_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW(),
  reviewed_by UUID REFERENCES users(id),
  accepted_by UUID REFERENCES users(id),
  shareedu_synced_at TIMESTAMPTZ, -- عند تثبيت المقعد
  notes TEXT
);

CREATE TABLE stage_history (
  id SERIAL PRIMARY KEY,
  application_id UUID REFERENCES applications(id) ON DELETE CASCADE,
  from_stage VARCHAR(20),
  to_stage VARCHAR(20) NOT NULL,
  changed_by UUID REFERENCES users(id),
  changed_at TIMESTAMPTZ DEFAULT NOW(),
  reason TEXT
);

CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email VARCHAR(120) UNIQUE NOT NULL,
  name VARCHAR(100) NOT NULL,
  role VARCHAR(20) NOT NULL, -- admin/admissions/viewer
  branch_id INT REFERENCES branches(id),
  is_active BOOLEAN DEFAULT TRUE,
  last_login_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE audit_logs (
  id BIGSERIAL PRIMARY KEY,
  actor_id UUID REFERENCES users(id),
  action VARCHAR(50) NOT NULL,
  entity_type VARCHAR(30),
  entity_id VARCHAR(50),
  metadata JSONB,
  ip_address INET,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE whatsapp_messages (
  id BIGSERIAL PRIMARY KEY,
  application_id UUID REFERENCES applications(id),
  direction VARCHAR(10), -- 'in' or 'out'
  message_type VARCHAR(20),
  content TEXT,
  status VARCHAR(20),
  sent_at TIMESTAMPTZ DEFAULT NOW()
);

⚡ للرجوع السريع: مبادرة فردية من الأستاذ عبدالله غراب لمعالجة الوضع الراهن — إذ بدأ فعليًا موسم القبول والتسجيل — فبادر بالتعاون المباشر (as we speak) مع فريق ShareEdu لتشغيل رحلة القبول والتسجيل كاملة مباشرة داخل بيئة ShareEdu نفسها، لا عبر أي من السيناريوهات الثلاثة أعلاه. جُرِّبت فعليًا، وهي الآن في مرحلة الإنهاء والإطلاق. له منّا كل التقدير على سرعة الاستجابة في هذا التوقيت الحرج. التفاصيل الكاملة للعملية في quick-solution.html ↗.

🔍 كيف يختلف هذا الحل تقنيًا عن السيناريوهات الثلاثة أعلاه؟

الفرق الجوهري: السيناريوهات الثلاثة في القسم 3 تتعامل جميعها مع سؤال "أين تُخزَّن البيانات خارج ShareEdu؟" (PostgreSQL مستقل / PostgreSQL موحّدة لدى ShareEdu / Firebase). الحل الحالي السريع يتجاوز هذا السؤال كليًا: لا قاعدة بيانات خارجية، ولا webhook، ولا جسر تكامل — العملية بأكملها (تسجيل ولي الأمر → نموذج الطالب → المقابلة → القبول → اختبار القبول) تُنفَّذ داخل تبويبات ShareEdu الإدارية القائمة أصلًا (إدارة التسجيل، إدارة المقابلات)، على نفس بنية Classic ASP + Access DB الحالية، دون أي طبقة وسيطة.

📌 آثار هذا التحديث على القرارات المطروحة في هذه الورقة

  1. لا يُلغي التوصية في القسم 4 (PostgreSQL مستقل) — الحل السريع يغطّي مسار القبول والتسجيل التشغيلي فقط، بينما تبقى الحاجة قائمة لأي طبقة بيانات مستقلة إن احتاج المشروع لاحقًا تحليلات، تقارير عبر الفروع، أو دمجًا مع أدوات قسم الاتصال (Chatbot، لوحات تحكم تسويقية).
  2. توضيح تنظيمي مهم: لا علاقة تنظيمية أو تقنية بين الأستاذ عبدالله غراب وتوصية PostgreSQL المستقلة أصلًا — الحل السريع مسار منفصل تمامًا عن مسار مبادرة #التحول_الرقمي_لمدارس_دار_الفرسان_2027 (المطوّر). لن تنشأ أي علاقة بينهما إلا إذا انضمّ الأستاذ عبدالله غراب رسميًا لمبادرة #التحول_الرقمي_لمدارس_دار_الفرسان_2027 مستقبلًا، عندها فقط يصبح التنسيق التقني بين المسارين موضوعًا مطروحًا.
  3. يُثبت عمليًا أن ShareEdu منفتحة على تكامل مباشر مع أطراف خارجية (الأستاذ عبدالله غراب) — معطى مفيد عند نقاش نقاط القسم 8 معهم مستقبلًا.
  4. سؤال تقني مفتوح يستحق التوثيق هنا للرجوع إليه: هل الحل السريع يكتب مباشرة على Access DB الخاصة بـ ShareEdu، أم عبر واجهة إدارية معزولة لا تلامس الجداول الأصلية؟ هذا يحدّد مدى تأثيره على القيود المعروفة لـ Access (2GB حجم أقصى، 10-15 مستخدمًا متزامنًا) المذكورة سابقًا في هذه الورقة — يُفضَّل تأكيدها مع الأستاذ عبدالله غراب مباشرة قبل الاعتماد الكامل على الحل في موسم قبول مزدحم.
  5. يُنصح بمواءمة الجدول الزمني بين إطلاق هذا الحل السريع وأي قرار مستقبلي بشأن السيناريوهات الثلاثة، تفاديًا لازدواجية العمل على نفس تدفّق القبول من مسارين مختلفين.

10. تحليل ضرورة الأطراف — بصدق ومسؤولية

⚠ ملاحظة صادقة: ليست كل الأطراف الثمانية بنفس درجة الضرورة الفعلية. التصنيف أدناه يميّز بين ما هو ضروري تشغيليًا وما هو مُضاف بناءً على طلب مباشر دون تقييم مستقل بأنه ضرورة قصوى — حتى تُتّخذ القرارات بعين مفتوحة، لا بافتراض تساوي الأدوار.

🟢 للسيناريو الأدنى (صفحة الهبوط المجانية)

يكفي فعليًا 4–5 أطراف نشطة: مركز صناعة القرار، القبول والتسجيل، المطوّر، ShareEdu، ولمسة خفيفة من الاتصال المؤسسي للهوية البصرية فقط (لا شراكة تنفيذية كاملة). باقي الأطراف لا تُشكّل عبئًا حقيقيًا على هذا المسار المحدود، ويمكن تجاهلها كليًا في هذه المرحلة.

🟡 للرؤية الشاملة (السيناريو 2 أو 4)

الثمانية جميعًا منطقية عند هذا الاتساع، لكن يُفضَّل الإبقاء على دمج "الإدارة الأكاديمية" و"الشؤون التعليمية" في طرف واحد (كما اعتمدنا فعليًا)، وإبقاء "إدارة الجودة" في مرتبة استشاري (Consulted) بدل مسؤول (Responsible) كامل في مصفوفة RACI — أي دوره أخف وزنًا من باقي الأطراف، يُستشار عند الحاجة فقط ولا يُعطَّل به سير العمل اليومي.

لماذا هذا التمييز مهم عمليًا؟

كثرة الأطراف الرسمية في مصفوفة قرار واحدة تخلق عبء حوكمة يتعارض مع الروح الأصلية للمبادرة — حل سريع وخفيف (صفحة هبوط بلا تعقيد). كل طرف إضافي بصفة "مسؤول" كامل يعني: توقيعات إضافية، احتمال تأخير أكبر في الموافقات، وتعقيد تنسيقي لا يتناسب مع بساطة المسار الأدنى. التمييز بين Responsible وConsulted ليس تفصيلًا شكليًا — إنه ما يحفظ سرعة الإنجاز.

14. 🔒 محرك قاعدة بيانات ShareEdu الحالي — وتوافقه مع رؤية الاتحاد (PostgreSQL)

🔒 للاطلاع الداخلي فقط (meandyou): امتداد تقني مباشر للقسم 13 أعلاه — لا يُدرَج بأي صيغة في أي ملف عام أو في الروبوت المشترك.

🗄️ ما هو محرك قاعدة البيانات الفعلي؟

ShareEdu تعتمد حاليًا على Microsoft Access، عبر محرك Jet/ACE (Jet للصيغة القديمة .mdb، وACE للصيغة الأحدث .accdb). هذا محرك قائم على ملف واحد (file-based) لا خادم قاعدة بيانات حقيقي:

  • كامل قاعدة البيانات = ملف واحد يجلس على القرص، لا يستمع على أي منفذ شبكي
  • صفحات ASP الكلاسيكية تتصل به عبر ADO بمزوّد OLEDB (Microsoft.Jet.OLEDB.4.0 أو Microsoft.ACE.OLEDB.12.0)
  • قيود معروفة: حد أقصى 2GB لحجم الملف، وتدهور أداء حاد بعد 10-15 اتصالًا متزامنًا (قفل على مستوى الملف، لا عميل-خادم حقيقي)

⚙️ آلية عملهم الحالية خطوة بخطوة (لكل طلب صفحة)

  1. المستخدم (ولي أمر أو موظف) يفتح صفحة .asp عبر المتصفح
  2. خادم IIS ينفّذ كود ASP الكلاسيكي (VBScript غالبًا) المُضمَّن بين وسوم <% %>
  3. الكود يفتح كائن اتصال ADO (Server.CreateObject("ADODB.Connection")) بمزوّد OLEDB لـ Access
  4. يُنفَّذ استعلام SQL عبر ADODB.Recordset مباشرة على ملف .accdb المخزَّن محليًا على قرص الخادم
  5. النتائج تُدرَج مباشرة داخل HTML الناتج عبر Response.Write أو الكتابة المباشرة بين وسوم ASP
  6. الصفحة الكاملة تُرسَل للمتصفح — لا توجد أي طبقة API أو REST وسيطة، الاتصال بقاعدة البيانات يحدث ضمن نفس دورة تنفيذ الصفحة مباشرة

هذا النمط (يُعرف تقنيًا بـ "Page-Level Data Access") هو سبب بساطة حل الأستاذ عبدالله غراب وسرعته: أي إضافة تحدث داخل نفس هذا النمط (صفحة ASP جديدة تتصل بنفس ملف Access) لا تحتاج أي بنية جديدة — بعكس أي محاولة لإدخال طبقة بيانات مختلفة الطبيعة.

⚖️ التوافق مع PostgreSQL — ثلاثة مستويات، لا اثنان فقط

في نقاشنا السابق (القسم 13) وصفت طرفين متطرفين فقط. لكن سؤال "الانفتاح مع اتحاد المكونات" يكشف عن مستوى ثالث أوسط — وهو الأقرب فعليًا لما تسأل عنه:

المستوى طبيعة العلاقة بـ Access تعاون ShareEdu المطلوب
1) اتحاد الكتلة المستقل (القسم 4 الأصلي) PostgreSQL مستقلة تمامًا وموازية، لا تلمس Access إطلاقًا — استقبال بيانات فقط عبر webhook أحادي الاتجاه لا يُذكر
2) 🆕 التعايش عبر ODBC (انفتاح جزئي) PostgreSQL تعمل بجانب Access، عبر جسر ODBC — التفاصيل أدناه محدود (تركيب + إذن نشر)
3) الانفتاح الكامل — استبدال المحرك ترحيل كامل (migration) لكل البيانات والاستعلامات إلى PostgreSQL كامل وحتمي

🆕 المستوى 2 بالتفصيل — "الانفتاح مع اتحاد المكونات"

هذا هو المسار الذي يجمع بين الاثنين فعليًا، عبر آليتين تقنيتين متاحتين في نفس نمط "Page-Level Data Access" الموصوف أعلاه، دون أي قطيعة فورية مع النظام الحالي:

أ) صفحات جديدة تتصل بـ PostgreSQL مباشرة: تركيب مشغّل psqlODBC على خادم IIS نفسه، وإنشاء DSN يشير لخادم PostgreSQL. أي صفحة .asp جديدة (مثل وحدات تحليلات أو تقارير) تفتح اتصالًا بصيغة Provider=MSDASQL;DSN=... بدل مزوّد Jet/ACE. الصفحات القديمة تبقى كما هي بلا أي تعديل، متصلة بـ Access كالمعتاد.

ب) الجداول المرتبطة (Linked Tables) داخل Access نفسها: تقنية أصلية في Access تجعله "يتظاهر" بأن جداول PostgreSQL هي جداول محلية داخل نفس ملف .accdb، فتصبح مرئية وقابلة للاستعلام بنفس صيغة SQL القديمة تقريبًا — رغم أن البيانات الفعلية تعيش في PostgreSQL. هذا أقرب مسار "تدرّجي" حقيقي لدمج الأنظمة، ويحتاج تعاون ShareEdu بحدّه الأدنى فقط (تركيب مشغّل ODBC + إذن ربط الجداول)، لا تسليم كامل الكود المصدري أو خادمهم.

🔧 متطلبات المستوى 3 التقنية (الترحيل الكامل)

تغيير مزوّد الاتصال من Microsoft.ACE.OLEDB إلى مشغّل ODBC رسمي (psqlODBC) مثبَّت على خادم IIS نفسه
ترحيل البيانات نقل كل السجلات من .accdb إلى PostgreSQL (عبر pgloader أو تصدير/استيراد CSV)
إعادة كتابة كل استعلامات SQL لهجة Access SQL تختلف عن PostgreSQL: IIF() ↔ CASE WHEN، TOP N ↔ LIMIT N، حقول Memo ↔ TEXT، عامل الدمج & ↔ || — كل استعلام داخل كل صفحة .asp يحتاج مراجعة يدوية
الوصول لكود ASP وخادم IIS يتطلب من فريق ShareEdu منح وصول فعلي لكودهم المصدري وخادمهم — هذا وحده يفسّر لماذا هذا المسار مستحيل عمليًا دون تعاونهم الكامل

15. 🔒 بدائل PostgreSQL للتعايش مع بنية ShareEdu (ADO + Jet/ACE)

🔒 للاطلاع الداخلي فقط (meandyou): امتداد مباشر للقسم 14 — لا يُدرَج بأي صيغة في أي ملف عام أو في الروبوت المشترك.

PostgreSQL ليست الخيار الوحيد لمستوى "التعايش عبر ODBC" (المستوى 2 في القسم 14). إليك خمسة بدائل حقيقية، مرتبة من الأقرب توافقًا مع البيئة الحالية إلى الأبعد:

البديل آلية الاتصال من ASP أداة ترحيل جاهزة؟ التكلفة
⭐ SQL Server Express OLEDB أصلي (MSOLEDBSQL) — نفس نمط الاتصال الحالي تمامًا، فرق حرفي واحد في connection string نعم — Access Upsizing Wizard (أداة رسمية من Microsoft مدمجة في Access) مجانية (حتى 10GB)
Azure SQL Database نفس OLEDB/ODBC القياسي، لكن الخادم سحابي بدل محلي نعم — نفس Upsizing Wizard + أدوات هجرة سحابية اشتراك سحابي (يبدأ من مستوى مجاني محدود)
PostgreSQL (المرجع السابق) ODBC عبر psqlODBC، أو جداول مرتبطة في Access لا أداة رسمية — ترحيل يدوي أو عبر pgloader مجانية بالكامل (مفتوحة المصدر)
MySQL / MariaDB ODBC عبر MySQL Connector/ODBC، دعم جداول مرتبطة مشابه لا أداة رسمية من Microsoft — أدوات طرف ثالث فقط مجانية بالكامل (مفتوحة المصدر)
SQLite ODBC عبر مشغّل طرف ثالث — لا يحل مشكلة التزامن أصلًا (لا يزال محرّكًا قائمًا على ملف واحد مثل Access) غير مناسب لهذا الغرض تحديدًا مجانية

⭐ لماذا SQL Server Express يستحق الأولوية في النقاش؟

هو الخيار الوحيد من نفس عائلة Microsoft التي تعمل بها البيئة الحالية بالكامل (IIS + Classic ASP + Windows Server). هذا يعني عمليًا:

  • لا "قفزة تقنية غريبة" — فريق ShareEdu على الأرجح مطّلع فعلًا على أدوات Microsoft، بعكس PostgreSQL/MySQL التي تتطلب تعلّم بيئة جديدة كليًا
  • أداة الترحيل مدمجة أصلًا في Access (Upsizing Wizard) — تنقل الجداول والعلاقات والمفاتيح تلقائيًا، وتنشئ الجداول المرتبطة بنفسها دون كتابة كود ترحيل يدوي
  • حد 10GB مجانًا كافٍ لسنوات عديدة مقارنة بحد الـ2GB الحالي في Access — يحل مشكلة الحجم الأصلية دون أي تكلفة
  • يبقى الخيار الأنسب لمستوى "الانفتاح الكامل" (المستوى 3) لاحقًا إن قرّرت ShareEdu ذلك — نفس المحرك يخدم مسار التعايش والاستبدال الكامل معًا، بعكس PostgreSQL التي تحتاج قرارًا منفصلاً لكل مستوى

🔗 الخلاصة العملية: توصية القسم 4 الأصلية بـ PostgreSQL كانت مبنية على سيناريو "الاتحاد المستقل" (المستوى 1) حيث لا صلة بـ Access أصلًا، فاختيار المحرك لم يكن مقيّدًا ببيئة ShareEdu. أما في سياق "التعايش" (المستوى 2) الذي نناقشه الآن، فـ SQL Server Express يبدو مرشّحًا أقوى تقنيًا وسياسيًا — يستحق طرحه صراحة عند أي نقاش مستقبلي مع فريق ShareEdu كبديل عملي لـ PostgreSQL.

2️⃣ الحقائق التقنية الجاهزة (لا حاجة لإعادة بحثها في الاجتماع)

البنية الحالية Classic ASP + Microsoft Access (محرك Jet/ACE) — ملف واحد، لا خادم عميل-خادم حقيقي. حد 2GB، وتدهور أداء بعد 10-15 اتصالًا متزامنًا (القسم 14)

البديل الأقوى تقنيًا/سياسيًا SQL Server Express — نفس عائلة Microsoft، أداة ترحيل رسمية مدمجة (Access Upsizing Wizard)، حد مجاني 10GB (القسم 15)
